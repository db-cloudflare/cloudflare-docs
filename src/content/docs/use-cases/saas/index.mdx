Build multi-tenant platforms with custom domains, isolated compute, and per-customer configuration.

SaaS platforms face unique challenges: supporting custom domains for customers, isolating tenant data and compute, and scaling infrastructure as your customer base grows. Cloudflare for Platforms provides the building blocks to extend Cloudflare's capabilities to your own customers.

This guide walks you through the Cloudflare solutions available for building and securing a SaaS platform.

Let your customers deploy their own code on your platform with isolated execution environments.

### Solutions

<CardGrid>

<LinkTitleCard title="Workers for Platforms" href="/cloudflare-for-platforms/workers-for-platforms/" icon="workers">

Let customers deploy their own code on your platform with isolated execution.

</LinkTitleCard>

<LinkTitleCard title="Dispatch Namespaces" href="/cloudflare-for-platforms/workers-for-platforms/configuration/dispatch-namespaces/" icon="workers">

Organize and route requests to customer Workers.

</LinkTitleCard>

</CardGrid>

### What you get

- **Tenant isolation** - Each customer's code runs in isolated V8 isolates
- **Custom logic** - Customers can extend your platform with their own code
- **Dispatch routing** - Route requests to the correct customer Worker
- **Observability** - Tail Workers capture logs across all tenant code

### Getting started with Workers for Platforms

<Details header="Set up Workers for Platforms">

1. **Create a dispatch namespace**:
   ```sh
   npx wrangler dispatch-namespace create my-dispatch-namespace
   ```

2. **Upload a user Worker** to the namespace:
   ```sh
   npx wrangler deploy --dispatch-namespace my-dispatch-namespace
   ```

3. **Create a dispatch Worker** to route requests:
   ```js
   export default {
     async fetch(request, env) {
       // Get customer identifier from request
       const customerId = request.headers.get("X-Customer-ID");
       
       // Get the customer's Worker from the dispatch namespace
       const userWorker = env.DISPATCH_NAMESPACE.get(customerId);
       
       // Forward the request to the customer's Worker
       return userWorker.fetch(request);
     }
   };
   ```

4. **Configure the dispatch binding** in `wrangler.toml`:
   ```toml
   [[dispatch_namespaces]]
   binding = "DISPATCH_NAMESPACE"
   namespace = "my-dispatch-namespace"
   ```

</Details>

For the complete guide, refer to [Workers for Platforms get started](/cloudflare-for-platforms/workers-for-platforms/get-started/).

</LinkTitleCard>

<LinkTitleCard title="Durable Objects" href="/durable-objects/" icon="durable-objects">

</LinkTitleCard>

</LinkTitleCard>

</CardGrid>

### What you get

- **Database per tenant** - Complete data isolation with D1
- **Object storage** - Store customer files with R2
- **Real-time coordination** - Durable Objects for stateful workflows
- **Edge configuration** - KV for tenant settings and feature flags

### Data isolation patterns

<Details header="Per-tenant database isolation">

Create separate D1 databases for each tenant:

```js
// Create a database for a new tenant
const db = await env.D1.prepare(
  "CREATE DATABASE IF NOT EXISTS tenant_" + tenantId
).run();

// Query tenant-specific database
const tenantDb = env[`DB_${tenantId}`];
const results = await tenantDb.prepare(
  "SELECT * FROM users WHERE active = ?"
).bind(true).all();
```

For shared database with row-level isolation:

```js
// Always include tenant_id in queries
const results = await env.DB.prepare(
  "SELECT * FROM users WHERE tenant_id = ? AND active = ?"
).bind(tenantId, true).all();
```

</Details>

<Details header="Per-tenant object storage">

Organize R2 storage by tenant:

**Option 1: Prefix-based isolation**
```js
// Store files with tenant prefix
await env.R2.put(`${tenantId}/uploads/${filename}`, file);

// List tenant files
const objects = await env.R2.list({ prefix: `${tenantId}/` });
```

**Option 2: Separate buckets per tenant**
- Create buckets programmatically via API
- Use bucket-level access controls
- Better for compliance requirements

</Details>

- **API protection** - Secure your platform APIs

### Security configuration

<Details header="Per-tenant rate limiting">

Configure rate limits that apply per customer:

1. Go to **Security** > **Application Security** > **Rate limiting rules**
2. Create a rule with:
   - **Expression**: `http.host contains "customer-domain.com"`
   - **Counting expression**: Use customer identifier header or cookie
   - **Rate**: Set appropriate limits per tenant tier

Example for API rate limiting:
```
(http.request.uri.path contains "/api/") and 
(http.request.headers["x-tenant-id"] eq "tenant-123")
```

</Details>

- **Performance insights** - Identify optimization opportunities

### Usage tracking

<Details header="Track per-tenant usage with Analytics Engine">

```js
export default {
  async fetch(request, env) {
    const tenantId = request.headers.get("X-Tenant-ID");
    const startTime = Date.now();
    
    // Process request
    const response = await handleRequest(request, env);
    
    // Record usage metrics
    env.ANALYTICS.writeDataPoint({
      blobs: [tenantId, request.cf.country],
      doubles: [Date.now() - startTime, response.headers.get("Content-Length")],
      indexes: [tenantId]
    });
    
    return response;
  }
};
```

Query usage for billing:
```sql
SELECT 
  blob1 as tenant_id,
  COUNT(*) as requests,
  SUM(double2) as bytes_transferred
FROM analytics
WHERE timestamp > now() - interval '30 days'
GROUP BY tenant_id
```

</Details>
